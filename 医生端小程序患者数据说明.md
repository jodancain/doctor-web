# 医生端小程序：患者列表与详情

本文说明「我的患者」工作台（`packages/consult/doctor-client`）如何拉取患者数据、常见空白原因及本次修复要点。

## 数据流

1. 前端通过 `miniprogram/utils/request.js` 调用后端接口 `/api/miniprogram`，`action` 为 `getDoctorPatients` / `getDoctorPatientOverview` / `getDoctorDashboardStats`。
2. 后端通过 JWT 解出的 `openid` 校验 `users` 集合中是否存在 **`role === 'doctor'`** 的记录；通过后再拉取患者列表。
3. 患者列表来源为 `users` 集合中 **非医生** 账号（`role !== 'doctor'`），兼容 `role` 为 `user`、`patient` 或历史未填写 `role` 的文档（旧逻辑仅 `role === 'user'` 会漏数据）。

## 本次已修复问题

### 1. 患者详情页路由错误

- **现象**：点击患者卡片无反应或提示页面不存在。
- **原因**：代码跳转到未在 `app.json` 注册的 `/pages/doctor/patient-detail/index`。
- **修复**：改为分包路径 **`/packages/consult/patient-detail/index?id=...`**。

### 2. 患者查询口径回归（本次修复重点）

- **现象**：数据库有患者，医生端列表仍为空。
- **原因**：`getDoctorPatients` / `getDoctorDashboardStats` 一度退回为 `where({ role: 'user' })`，部分账号为其它枚举或未带 `role`。
- **修复**：统一改为“排除医生 + 必须有 `_openid`”口径，兼容历史数据并与统计接口保持一致。

### 3. 调试账号 `doctoradmin`

- **现象**：使用 `doctoradmin` + `adminadmin` 本地 Mock 登录时，列表长期为空或 403。
- **原因**：本地存储的医生信息与云函数侧 **真实 OPENID** 不一致，云侧可能校验失败。
- **修复**：在 `getDoctorPatients` 失败且当前用户为 `doctoradmin` 时，前端展示 **演示患者** 列表，并提示使用真实医生账号登录以联调云端数据。

### 4. 其它

- **高危预警**：原 `/pages/doctor/alerts` 未注册，已改为 Toast 提示，避免无效跳转。
- **患者详情**：`bmi` 为空时避免 `toFixed` 报错。

## 部署与自检

1. 修改后端后务必重启服务并发布最新镜像（若使用 Docker）。
2. 在云开发控制台确认 `users` 中存在：
   - 至少一条当前微信号对应的 **`role: 'doctor'`** 记录（医生用账号密码登录并绑定 OPENID）；
   - 若干 **`role` 为患者侧**（如 `user`）或 **非 doctor** 的账号。
3. 切换到底部「患者」Tab 会 **自动重新拉取** 列表。
