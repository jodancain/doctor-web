# 对话功能现状与优化建议

## 1. 现有实现快照
- **前端会话页**：`pages/chat/index.js` 通过 5s 轮询 `fetchConversation` 拉取最近 100 条消息，发送文本/图片直接调用 `sendMessage` 并将返回值追加列表，缺少消息状态管理与分页。【F:miniprogram/pages/chat/index.js†L18-L117】【F:miniprogram/pages/chat/index.js†L141-L194】
- **会话列表页**：`pages/doctor-messages/index.js` 以 10s 轮询刷新最近 20 条会话，删除会话仅调用后端批量删除，无未读数或实时更新能力。【F:miniprogram/pages/doctor-messages/index.js†L1-L83】
- **后端接口**：云函数 `chat-service` 目前仅支持：
  - `fetchConversation`：双向查询并一次性返回 100 条消息，不支持分页或增量拉取。【F:cloudfunctions/healthDataService/src/services/chat-service.js†L1-L51】
  - `sendMessage`：简单写入消息，缺少会话 ID、状态、读回执等字段。【F:cloudfunctions/healthDataService/src/services/chat-service.js†L53-L82】
  - `listInbox`：按最新消息分组，但没有未读统计、已读同步或会话静态信息缓存。【F:cloudfunctions/healthDataService/src/services/chat-service.js†L84-L170】

## 2. 痛点与差距
- **实时性不足**：前端固定 5–10s 轮询，活跃对话体验落后于 WebSocket/长轮询的主流方案。
- **消息状态缺失**：没有“发送中/失败/重试/已读”状态，无法做本地乐观更新和失败兜底，图片上传也缺乏重试队列。
- **历史加载与缓存缺位**：一次性取 100 条，既浪费流量又缺少分页与本地缓存，首屏与回溯体验不佳。
- **数据模型简单**：消息缺少 `conversationId` 索引、`status`、`isRead`、`deviceId` 等字段；会话表未独立，导致未读数/草稿/置顶等能力难以扩展。
- **安全与稳定性**：缺少重复提交幂等、消息去重、文件安全校验（图片 MIME/大小）和异常监控。

## 3. 优化蓝图（对标头部 IM 方案）
### 3.1 连接与同步
- 引入 **WebSocket**（`wx.connectSocket`）作为首选通道，提供 `onMessage` 增量推送；兼容后台/弱网场景时退化为自适应长轮询（活跃 1s、静默 5–10s）。
- 建立 **增量同步协议**：客户端保存 `lastMessageId/lastAckTime`，后端提供 `syncMessages(sinceId, limit, direction)`，支持上拉分页与下拉增量。
- 为每条消息生成 **全局有序 ID**（如时间戳+随机或雪花），避免单设备时间漂移导致排序异常。

### 3.2 消息可靠性
- 前端实现 **乐观更新 + 状态机**：发送时先入本地队列（`pending`），成功改 `success`，失败标记 `fail` 并提供一键重试；图片上传失败自动重试 2–3 次并支持“重新上传”。
- 后端增加 **幂等与去重**：接受 `clientMsgId`，如已存在则直接返回，避免重复写入；为图片/文件做 MIME、大小白名单校验并在云存储启用防盗链。
- 引入 **消息回执**：`delivered/read` 状态更新，前端在对话列表和会话页展示未读角标和已读标记。

### 3.3 数据与性能
- 定义 **会话表 (`conversations`)**：维护 `conversationId (sorted_openid)`、`lastMessage`、`lastTime`、`unreadCount`、`draft`、`pinned` 等字段，`listInbox` 直接读该表减少聚合开销。
- 消息表增加 **复合索引**：`conversationId + createdAt`，`toOpenid + isRead`，提升分页和未读统计性能。
- **本地缓存**：首屏优先读 `wx.setStorage` 的最近 20 条，后台静默同步增量并合并；图片消息缓存 `tempFileURL`，减少重复获取。
- **资源加载优化**：发送/接收图片前生成压缩图与原图双通道；长列表使用虚拟列表或分段渲染避免卡顿。

### 3.4 交互体验
- 输入区支持 **多行自动伸缩、键盘避让**、表情/图片入口、发送按钮状态禁用；长按菜单提供复制、删除、重试。
- **时间分组与锚点**：在消息列表按 5–10 分钟显示一次时间戳；保留用户滚动位置，避免刷新跳动。
- **弱网提示与重连提示**：显示连接状态（已连接/重连中/离线）；发送失败展示红色感叹号可点重试。
- **辅助能力**：输入框草稿缓存，医生端可快速插入快捷回复；支持系统提示气泡（如已读/撤回/风险提示）。

### 3.5 监控与灰度
- 接入云监控埋点：发送成功率、平均延迟、重试率、上传失败率、WS 连接成功率；异常日志写入专用集合便于回溯。
- 灰度发布：先在内测账号开启 WebSocket + 增量同步，收集指标后再全面切换；提供开关快速回退到轮询模式。

## 4. 落地建议（迭代路径）
1. **协议层**：补充 `conversationId` 字段及索引，新增 `clientMsgId/status/isRead`，实现 `syncMessages` 接口；前端维护乐观状态与重试逻辑。
2. **连接层**：引入 WebSocket 客户端封装，带心跳与断线重连，保留自适应轮询兜底。
3. **存储层**：落地会话表与未读计数，消息表分页（`limit`/`skip` 或 `startAfter`）；消息列表支持上拉加载更多与本地缓存。
4. **体验层**：优化输入区、时间分组、长按菜单与失败重试；在会话列表展示未读数、最新状态和图片占位。
5. **安全与合规**：上传前做图片类型/大小校验，敏感词过滤，云函数加入幂等与异常告警；定期清理孤立文件与 mock 会话。

## 5. 本次性能优化分析（对话加载流畅性）
- **首屏加载平滑度**：在进入会话页时先尝试从 `wx.getStorage` 读取该会话的缓存消息，并立即落地到列表，再触发网络刷新，避免首屏空白与闪烁。【F:miniprogram/pages/chat/index.js†L16-L20】【F:miniprogram/pages/chat/index.js†L94-L117】
- **按需持久化策略**：网络拉取的新消息在合并后落地本地缓存，仅截取最近 100 条，既维持按需注入（未触达页面不加载其他会话数据），也减少后续切换/重开页面时的渲染压力。【F:miniprogram/pages/chat/index.js†L118-L175】【F:miniprogram/pages/chat/index.js†L249-L283】
- **加载状态优化**：初始化时不再强制清空本地消息，而是仅开启 `loading` 标记，结合缓存回显和滚动锚点避免多余的重排，提高弱网下的感知流畅度。【F:miniprogram/pages/chat/index.js†L67-L90】
