# 病例夹 (Medical Folder) 设计分析与转化方案

本文档旨在分析现有小程序项目的“病例夹”功能，对比参考项目 (`gout-butler-taro`) 的设计理念与数据模型，并提出后续的优化与转化方案。

## 1. 项目现状分析 (Current State)

### 1.1 现有架构 (`miniprogram/pages/health-info`)
目前的小程序“病例夹”模块主要作为一个**静态健康档案 (Electronic Health Record)** 存在，侧重于记录患者的病史与基础信息。

*   **核心流程**:
    *   **初始化 (`init`)**: 引导用户录入姓名、性别、出生日期、确诊年份。
    *   **仪表盘 (`index`)**: 采用“十二宫格”布局，涵盖基本信息、病史、既往史、生活史、过敏史等 12 个维度。
    *   **详情页 (`detail`)**: 提供文本或标签形式的增删改查功能。

*   **数据模型**:
    *   数据直接存储在 `users` 集合的根节点下。
    *   字段包括：`name`, `gender`, `birthDate`, `diagnosisYear`, `history` (现病史), `pastHistory` (既往史), `allergies` (过敏史数组) 等。

### 1.2 设计风格
*   **UI 风格**: 近期已对 `init` 页面进行了现代化改造（大圆角、渐变按钮、卡片式选择器）。
*   **交互**: 基础的表单交互与页面跳转。

---

## 2. 参考项目分析 (`gout-butler-taro`)

参考项目基于 Taro + React + Tailwind CSS，其设计理念更偏向于**动态健康管理 (Health Management)**。

### 2.1 数据模型 (`src/types.ts`)
参考项目定义了更结构化的 `PatientProfile`：

```typescript
export interface PatientProfile extends User {
  age: number;                  // 年龄 (现有项目存的是 birthDate)
  gender: 'Male' | 'Female';
  diagnosisDate: string;        // 确诊日期
  currentUricAcid: number;      // 当前尿酸值 (核心指标)
  targetUricAcid: number;       // 目标尿酸值 (管理目标)
  riskLevel: 'Low' | 'Medium' | 'High'; // 风险等级 (分层管理)
  lastAttack?: string;          // 最近发作时间
}
```

### 2.2 设计理念
*   **视觉语言**:
    *   使用 **Teal/Emerald (#0f766e)** 作为主色调，强调医疗与健康感。
    *   大量使用 **圆角卡片 (`rounded-2xl`)** 和 **柔和阴影 (`shadow-xl`)**。
    *   **色彩编码**: 饮食(橙)、饮水(蓝)、用药(紫)、发作(红) 等功能区隔明显。
*   **功能布局**:
    *   虽然 `profile` 和 `records` 页面在代码中为 `PlaceholderPage`，但从 `Home` 页可以看出，用户信息不仅是档案，更是驱动首页仪表盘（尿酸状态、AI 建议）的核心数据源。

## 3. 转化与优化方案 (Transformation Plan)

为了将现有的“病例夹”提升至参考项目的水平，建议执行以下转化方案：

### 3.1 数据层转化 (Data Transformation)

| 现有字段 | 建议字段/改动 | 原因/用途 |
| :--- | :--- | :--- |
| `birthDate` | 保持，前端动态计算 `age` | `age` 用于医学评估，`birthDate` 用于存档 |
| `diagnosisYear` | 升级为 `diagnosisDate` | 精确到月或日，便于计算病程 |
| (无) | **新增** `targetUricAcid` | 设定治疗目标（如 360 或 300），用于首页仪表盘展示达标情况 |
| (无) | **新增** `riskLevel` | 根据病史自动评估风险等级，指导 AI 建议 |
| `history` (纯文本) | 结构化拆解 (可选) | 将长文本拆分为“主要症状”、“发作频率”等结构化字段 |

### 3.2 UI/UX 转化 (Design Transformation)

#### 3.2.1 仪表盘升级 (`pages/health-info/index`)
*   **当前**: 简单的图标 + 文字宫格。
*   **目标**:
    *   **顶部卡片**: 增加用户“健康画像”概览卡片，展示 `风险等级`、`目标尿酸`、`病程` 等关键标签。
    *   **宫格优化**: 将图标背景改为参考项目中的 **色彩编码系统**（如过敏史用红色系，生活史用绿色系），并增加微渐变效果。

#### 3.2.2 详情页重构 (`pages/health-info/detail`)
*   **当前**: 简单的列表或文本框。
*   **目标**: **卡片式表单**。
    *   参考 `init` 页面的风格，将输入框、选择器封装在圆角卡片中。
    *   对于“过敏史”等标签页，使用更有质感的 `Chip` 组件（带删除图标和背景色）。

### 3.3 功能模块映射 (Mapping)

| 参考项目模块 | 现有小程序对应 | 转化动作 |
| :--- | :--- | :--- |
| `PatientHome` | `pages/home` | 首页已包含快捷入口，需强化“今日状态”卡片样式 |
| `Profile` | `pages/health-info` | **重点优化对象**，作为完整的电子病历本 |
| `Records` | `packages/records/*` | 现有记录功能较分散，可通过病例夹进行聚合索引 |

## 4. 实施路线图

1.  **P0 (已完成)**: `init` 页面 UI 重构，对齐参考项目风格。
2.  **P1 (进行中)**: `health-info/index` 仪表盘样式升级，引入健康画像卡片。
3.  **P2**: 数据结构扩展，在 `users` 集合中增加 `targetUricAcid` 等管理字段。
4.  **P3**: `detail` 详情页 UI 统一化，废弃原生表单样式，全面采用定制组件。

此方案旨在保留现有小程序完整功能逻辑的基础上，通过视觉和数据结构的升级，达到参考项目 `gout-butler-taro` 的现代化体验。
