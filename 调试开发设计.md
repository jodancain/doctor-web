# 调试开发设计文档

## 1. 引言
为了提高本地开发效率，减少对云资源的依赖，并在无网络环境下也能进行界面逻辑调试，本项目引入了本地调试账户机制。本机制允许开发者通过特定的预设账户直接登录系统，绕过云函数鉴权流程，获取模拟的用户身份数据。

## 2. 调试账户配置

系统预设了两个调试账户，分别对应“医生”和“患者”两种角色。

### 2.1 医生账户
- **账号**: `doctoradmin`
- **密码**: `adminadmin`
- **角色**: `doctor`
- **模拟数据**:
  ```javascript
  {
    _id: 'debug_doctor_id_001',
    _openid: 'debug_doctor_openid_001',
    username: 'doctoradmin',
    nickName: '调试医生',
    role: 'doctor',
    title: '主任医师',
    hospital: '本地调试医院',
    badge: '认证专家'
  }
  ```

### 2.2 患者账户
- **账号**: `patientadmin`
- **密码**: `adminadmin`
- **角色**: `user`
- **模拟数据**:
  ```javascript
  {
    _id: 'debug_patient_id_001',
    _openid: 'debug_patient_openid_001',
    username: 'patientadmin',
    nickName: '调试患者',
    role: 'user'
  }
  ```

## 3. 实现原理

### 3.1 登录拦截
在 `miniprogram/services/user.js` 服务层中，对 `login` 方法进行了增强。

1.  **拦截判断**: 在发起云函数调用前，检查传入的 `username` 是否存在于 `DEBUG_ACCOUNTS` 配置表中，且 `password` 是否匹配预设的通用密码 (`adminadmin`)。
2.  **Mock 响应**: 如果匹配成功，直接返回预定义的 JSON 对象，模拟云函数的成功响应结构。
3.  **透传处理**: 如果不匹配，则继续按照原有流程调用 `wx.cloud.callFunction` 请求后端。

### 3.2 数据隔离
- 调试账户的 `_openid` 均以 `debug_` 开头，与真实用户的 OpenID 格式区分。
- 建议在云函数端（如果需要更深度的 Mock 支持）也可以根据此特征进行数据隔离或 Mock 数据返回，防止污染生产环境数据库。

## 4. 使用场景
- **UI 还原**: 快速查看不同角色的界面布局，无需频繁注册新号。
- **逻辑调试**: 调试与用户身份绑定的前端逻辑（如 tabbar 切换、权限控制）。
- **离线开发**: 在无网络或云服务配额受限时进行基础功能开发。

## 5. 扩展指南
如果需要添加更多测试场景（如“未认证医生”、“新注册用户”），可直接在 `miniprogram/services/user.js` 的 `DEBUG_ACCOUNTS` 对象中添加新的键值对配置即可。
