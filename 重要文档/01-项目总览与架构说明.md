# 01 — 项目总览与架构说明

## 一、项目定位

**痛风管家 (GoutCare)** 是一款面向痛风/高尿酸血症患者的慢病管理微信小程序，配套医生端 Web 管理后台。

### 核心价值
- 帮助患者记录尿酸值、痛风发作、用药、饮水、运动、饮食等健康数据
- 提供 AI 智能助手（"小风"）进行日常健康咨询
- 医生可通过 Web 后台查看患者数据、进行远程管理、AI 辅助临床决策
- 提供健康科普文章与食物嘌呤数据库

## 三、架构演进历史

| 阶段 | 时间 | 架构 | 说明 |
|------|------|------|------|
| V1 | 2025-11 | 微信云开发 (CloudBase) | 云函数 + 云数据库，零运维 |
| V2 | 2025-11 | Express + MongoDB (自建服务器) | 脱离云开发限制，全功能自主控制 |
| V2.1 | 2026-03-26 | 安全加固 | JWT加密、bcrypt、限流、安全头 |
| V2.2 | 2026-03-27 | 腾讯元器 AI 接入 | 替换阿里通义千问 + Google Gemini |
| V2.3 | 2026-03-28 | AI 增强 + Open API | 多Provider框架、健康上下文注入、外部AI接口 |

### V1→V2 迁移关键决策
- **原因**：云开发免费额度受限、无法自定义域名、AI 接入不灵活
- **方案**：保留小程序前端不变，后端从云函数迁移到 Express，数据库从云数据库迁移到自建 MongoDB
- **兼容层**：`server/services/miniprogram/` 目录下的 `.cjs` 文件模拟了云函数的调用接口，使小程序端改动最小化
- **旧代码归档**：`_archived_cloudfunctions/` 目录保存了原始云函数代码

## 五、数据流示意

### 患者端请求流程
```
小程序页面 → services/*.js → utils/request.js
    → POST https://yundongyl.cn/api/miniprogram
    → Nginx (SSL终止) → Express → miniprogram.ts
    → 根据 action 字段分发到对应 service
    → MongoDB 查询/写入 → 返回 { code, msg, data }
```

### 医生端请求流程
```
React 页面 → fetch/axios → POST/GET https://yundongyl.cn/api/*
    → Nginx → Express → requireAuth 中间件（JWT校验）
    → 对应 route handler → MongoDB → 返回 JSON
```

### AI 对话流程（患者端 - 非流式）
```
小程序 ai-chat → POST /api/miniprogram { action: 'askAI' }
    → ai-service.cjs → 组装健康上下文 → POST yuanqi.tencent.com
    → 等待完整响应 → 返回 { reply: "..." }
```

### AI 对话流程（医生端 - SSE 流式）
```
React AIChat → POST /api/ai/chat { message, history }
    → ai.ts → https.request(yuanqi, { stream: true })
    → 逐 chunk 代理 SSE → 前端实时渲染
```

## 六、关键设计决策记录

| 决策 | 选择 | 原因 |
|------|------|------|
| 小程序后端通信 | 统一 POST /api/miniprogram + action 字段 | 最小化小程序端改动（原云函数就是单入口） |
| 数据库 | MongoDB（非 MySQL） | 文档模型天然适配健康记录的灵活字段 |
| AI 服务 | 腾讯元器（主）+ Provider 框架 | 元器内置医疗知识；框架支持无缝切换 |
| 认证 | JWT (HttpOnly Cookie + Bearer) | 小程序用 Bearer，Web 用 Cookie |
| 部署 | Docker Compose | 一键部署、环境隔离、易迁移 |
| SSL | Let's Encrypt + Certbot | 免费、自动续期 |
| 文件上传 | 本地磁盘 + Nginx 静态服务 | 简单直接，V3 可迁移到 OSS |
